LoRA fine-tuning · sequence-level distillation · held-out evaluation

Teaching a 0.5B model a private search protocol

SidSearch is a reproducible fine-tuning experiment that freezes a small base language model, trains only LoRA adapter weights on a private structured task, and measures before/after performance on the same held-out benchmark.

Headline Result

Base closed-book composite 0.2542 Untouched Qwen2.5-0.5B-Instruct
LoRA closed-book composite 0.3053 Same base model plus trained adapter
Trainable parameter fraction 0.2184% 1,081,344 trainable parameters

On 50 held-out SidSearch examples, the LoRA adapter improved composite score by 0.0511 absolute. The paired bootstrap 95% interval is [-0.0146, 0.1207], so this is a directional small-benchmark result, not a settled statistical claim.

The adapter improved task semantics such as intent, source, entity extraction, and clarification handling. It did not solve robust structured output: JSON validity dropped from 0.9600 to 0.8000, and schema compliance plus exact full-record match remain at zero in this run.

The open-book base row is a diagnostic artifact. In that run, the model often copied or continued the protocol prompt and produced truncated or repeated text instead of one complete JSON object, so it should not be read as a meaningful open-book capability result. Raw generations are written to results/base_open_book_results.json when the base benchmark script is run.

Task Definition

SidSearch converts a natural-language search request into one JSON execution plan. The protocol defines exact routing and formatting behavior for documents, email, GitHub, combined search, dates, missing information, and clarification. 

Input:
Find open GitHub issues about ReleaseRadar from last month

Expected output:
{
  "intent": "repository_search",
  "entities": ["ReleaseRadar"],
  "source": "github",
  "filters": {
    "start_date": "2026-06-01",
    "end_date": "2026-06-30",
    "owner": "Morgan",
    "file_type": null,
    "status": "open"
  },
  "rewritten_query": "ReleaseRadar open issues",
  "confidence": "high",
  "clarification_required": false,
  "clarification_question": null,
  "applied_rules": ["SS-001", "SS-002", "SS-012", "SS-015", "SS-018", "SS-019", "SS-020"]
}

Protocol Coverage

  • exact entity preservation
  • relative date resolution
  • explicit date filters
  • source routing
  • ambiguous source handling
  • missing entity handling
  • query expansion
  • quoted phrase preservation
  • freshness handling
  • negation and exclusion filters
  • repository, email, document, and multi-source routing
  • confidence assignment
  • clarification policy
  • file-type and status inference
  • output schema enforcement

System Design

1. ProtocolDefine the private SidSearch behavior and schema.
2. ScenariosGenerate structured task examples across rule categories.
3. TeacherUse Ollama qwen2.5:3b for sequence-level labels.
4. ValidateReject malformed JSON, invalid schema, and rule mismatches.
5. TrainFreeze the base model and update LoRA adapters only.
6. BenchmarkCompare base vs adapter on 50 held-out examples.
LayerPurpose
ProtocolDefines the target behavior before training.
Data pipelineCreates, validates, and splits examples.
Training pipelineInjects LoRA, freezes base weights, verifies gradients, trains adapters.
Evaluation pipelinePreserves raw outputs, parses JSON, computes metrics, reports failures.

Distillation Method

This project uses sequence-level distillation. The teacher receives the SidSearch protocol and one scenario, then emits a complete JSON target answer. The student is trained with supervised fine-tuning on accepted target sequences. 

teacher provider: Ollama
teacher model:    qwen2.5:3b
temperature:      0
method:           sequence-level distillation

This is not logit distillation. No teacher token probabilities, logits, or hidden states are transferred.

Evaluation Metrics

The benchmark runner records raw generations before parsing. Metrics include JSON validity, schema compliance, exact intent accuracy, source accuracy, confidence accuracy, clarification accuracy, entity F1, applied-rule F1, filter-field accuracy, exact full-record match, latency, and composite score. 

composite =
  0.20 * intent_accuracy
+ 0.10 * source_accuracy
+ 0.20 * filter_accuracy
+ 0.10 * clarification_accuracy
+ 0.15 * applied_rule_f1
+ 0.10 * json_validity
+ 0.10 * schema_compliance
+ 0.05 * entity_f1

The composite score improved, but strict schema reliability did not. A stronger next iteration would add constrained decoding, JSON repair, stricter assistant-token masking, more accepted teacher rows, or a schema-specific post-processor.

Claim Boundary

Supported claim: on 50 held-out SidSearch examples, the LoRA-adapted 0.5B model directionally improved composite task score from 0.2542 to 0.3053 while training only 0.2184% of parameters. The strict-output failures remain unresolved and are part of the result.

  • No claim of general model improvement.
  • No claim that the model is better than larger general-purpose models.
  • No claim of production readiness.
  • No claim that the adapter improves tasks outside SidSearch.
  • No claim that training-set performance demonstrates generalization.
