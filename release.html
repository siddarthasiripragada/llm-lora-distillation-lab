This page explains the SidSearch experiment from first principles: what LoRA is, why only a tiny fraction of parameters were trained, what distillation means, how the benchmark works, and what the result does and does not prove.

The Plain-English Version

A language model is a large mathematical system with many learned numbers called parameters. In this experiment, the base model had 495,114,112 parameters. Instead of retraining all of them, the experiment trained only 1,081,344 extra LoRA parameters.

Total model parameters 495.1M Base Qwen2.5-0.5B-Instruct model
Trainable LoRA parameters 1.08M Adapter weights updated during training
Trainable fraction 0.2184% 99.78% of the base model stayed frozen

The central question was simple: can a compact 0.5B model learn a private search-planning protocol it has never seen, without retraining the whole model?

What The Model Had To Learn

The task is called SidSearch. It converts natural-language requests into structured JSON execution plans across documents, email, GitHub, and multi-source search. 

User request:
Find open GitHub issues about ReleaseRadar from last month

Expected structured plan:
{
  "intent": "repository_search",
  "entities": ["ReleaseRadar"],
  "source": "github",
  "filters": {
    "start_date": "2026-06-01",
    "end_date": "2026-06-30",
    "owner": null,
    "file_type": null,
    "status": "open"
  },
  "rewritten_query": "ReleaseRadar open issues",
  "confidence": "high",
  "clarification_required": false
}

This is harder than just answering a question. The model has to follow a private schema, choose the right source, preserve entities, resolve relative dates, set filters, and avoid inventing unsupported fields.

What LoRA Means

LoRA stands for Low-Rank Adaptation. It is a parameter-efficient fine-tuning method. Instead of changing the original model weights, LoRA adds small trainable matrices inside selected model layers.

ConceptPlain meaningIn this experiment
Base model The original pretrained model. Qwen/Qwen2.5-0.5B-Instruct
Frozen weights Weights that do not change during training. The original 0.5B model remained frozen.
Adapter A small trainable add-on attached to the model. LoRA matrices were trained for SidSearch.
Rank A control for how much capacity the adapter has. Rank 8.
Target modules The parts of the model where LoRA is inserted. q_proj, k_proj, v_proj, o_proj

A useful mental model: the base model is the general engine, and the LoRA adapter is a small steering kit for one specific road. The engine stays intact; the adapter nudges behavior toward the target task.

What Distillation Means Here

Distillation means using a stronger or more capable system to create training examples for a smaller model. In this project, a local teacher model generated JSON target answers from the SidSearch protocol and synthetic scenarios.

1. ProtocolDefine SidSearch rules and schema.
2. ScenariosCreate fictional search requests.
3. TeacherUse Ollama qwen2.5:3b to draft answers.
4. ValidateKeep only outputs that pass strict checks.
5. StudentTrain the LoRA adapter on accepted examples.

The final training data was not blindly trusted. Outputs were rejected when JSON was malformed, schema fields were wrong, enums were invalid, or deterministic fields disagreed with the rule engine.

Experiment Setup

Dataset Split

  • 150 training examples
  • 25 validation examples
  • 50 held-out benchmark examples
  • 204 accepted teacher rows
  • 71 rejected teacher rows

Training Setup

  • Base model: Qwen2.5-0.5B-Instruct
  • Teacher model: Ollama qwen2.5:3b
  • LoRA rank: 8
  • LoRA alpha: 16
  • Trainable parameters: 0.2184%

The held-out benchmark was frozen before final evaluation. That matters because the final score should measure generalization to examples the adapter did not train on.

Result

The composite score moved from 0.2542 to 0.3053. That is an absolute gain of 0.0511, or about 20.1% relative improvement. 

base closed-book composite: 0.2542
LoRA closed-book composite: 0.3053
absolute gain:              0.0511
relative gain:              ~20.1%
paired bootstrap 95% CI:    [-0.0146, 0.1207]
JSON validity change:       0.9600 -> 0.8000
full-record match:          0.0000 -> 0.0000

The adapter improved partial task behavior: intent detection, source selection, entity extraction, and private task understanding. That means it became more aligned with the SidSearch task.

The important caveat: robust structured output was not solved. JSON validity dropped after LoRA, and exact full-record match stayed at zero. The confidence interval is also wide because the held-out benchmark has only 50 examples.

How The Composite Score Is Calculated

A composite score is a weighted summary of several metrics. It is useful for a single headline number, but the individual metrics explain what actually improved or failed. 

composite =
  0.20 * intent_accuracy
+ 0.10 * source_accuracy
+ 0.20 * filter_accuracy
+ 0.10 * clarification_accuracy
+ 0.15 * applied_rule_f1
+ 0.10 * json_validity
+ 0.10 * schema_compliance
+ 0.05 * entity_f1

The headline gain came from partial-credit metrics. It should not be read as proof that the model always emits a correct execution plan.

What This Proves And What It Does Not

SupportedNot supported
A small frozen model can be nudged toward a narrow private task using LoRA. The model is production ready.
Only 0.2184% of parameters needed to be trainable for measurable task movement. The model became generally smarter.
Partial semantic metrics improved on the held-out benchmark. The model reliably emits perfect JSON plans.
The pipeline verifies frozen weights, LoRA gradients, leakage checks, and before/after benchmarks. The result will transfer automatically to unrelated tasks.

The real takeaway is balanced: parameter-efficient fine-tuning can specialize a compact model, but strict structured generation still benefits from constrained decoding, JSON repair, better supervision, or a schema-specific post-processor.
